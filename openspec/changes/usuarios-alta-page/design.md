# Design: Usuarios — Dedicated Create Page, DNI, Required Name, Password Confirmation

## Technical Approach

Additive, backwards-compatible extension of the `usuarios-crud` module. Backend: add a nullable `dni` column with a real MySQL `UNIQUE` index, mirror the existing `username` two-layer uniqueness pattern (`DUPLICATE_*_ERROR` constant + `findUnique`/`findFirst` pre-check + `P2002` catch) for `dni`, and tighten `nombre`/`apellido` DTO validators. Frontend: extract `create` into a dedicated route `/usuarios/nuevo` under the existing `(dashboard)` route group (no new layout), keep `edit` inline on the list page, add a DNI table column and a client-only confirm-password gate. No structural/architectural departure from `usuarios-crud`'s design — this reuses its module shape, its lib wrapper conventions, and its Tailwind styling verbatim.

## DB Engine Finding (drives the unique-constraint decision)

`server/prisma/schema.prisma` `datasource db` → `provider = "mysql"`. **Engine is MySQL.**

**Correction to a common misconception:** MySQL (InnoDB) permits an **unlimited** number of `NULL` values in a `UNIQUE` index. This is standard SQL behavior — two `NULL`s are never considered "equal", so they never collide. The claim that "MySQL allows only ONE null in a unique column" is **false** for InnoDB; that behavior does not apply here.

Consequence: every legacy `User` row (all of which will have `dni = NULL` immediately after the migration, since there is no backfill) coexists freely under a `UNIQUE` index on `dni`. The index enforces uniqueness **only among non-null values** — exactly the semantics the proposal wants. There is therefore **no risk** of the DB-level unique constraint rejecting a second (or Nth) null row.

## Decision: Keep a real DB `@unique` on the nullable `dni` (do NOT drop to app-layer-only)

**Choice**: `dni String? @unique` in the Prisma schema, producing a MySQL `UNIQUE INDEX User_dni_key`.

**Rationale**: Because the engine is MySQL and MySQL allows multiple nulls under a unique index, the DB constraint is safe to add and gives us the same TOCTOU backstop `username` already relies on: the `findUnique` pre-check is not atomic with `create`, so two concurrent requests with the same new DNI can both pass the pre-check; the DB `UNIQUE` constraint is the real serialization point and its `P2002` violation is translated back into the same `409`. Dropping the DB constraint (the proposal's fallback "app-layer only" option) would only be justified if the engine rejected multiple nulls — it does not — and doing so would silently reintroduce the exact TOCTOU race that `usuarios-crud` deliberately closed for `username` (commit `0e887bc`). So the proposal's primary plan ("DB `@unique` + `P2002` catch as a TOCTOU backstop") stands unchanged.

**Rejected alternative — app-layer-only (no DB constraint)**: would be correct on Postgres-partial-index paranoia or if legacy nulls collided, but on MySQL it buys nothing and loses the concurrency backstop. Not chosen. (Recorded here so a future reader on a different engine knows the decision was engine-specific, not dogmatic.)

## 1. Prisma Schema + Migration

### Schema change (`server/prisma/schema.prisma`)

Add one field to `model User` (place it near `passwordHash`, before `nombre`):

```prisma
dni          String?  @unique
```

`nombre`/`apellido` **stay `String?`** — no `NOT NULL` change (see the app-layer-only decision inherited from the proposal). Required-ness is enforced at the DTO/form layer only.

### Migration (`server/prisma/migrations/<timestamp>_add_user_dni/migration.sql`)

Generated by `npx prisma migrate dev --name add_user_dni`. Expected content (mirrors the format of `20260708030803_add_user_status_creator`):

```sql
-- AlterTable
ALTER TABLE `User` ADD COLUMN `dni` VARCHAR(191) NULL;

-- CreateIndex
CREATE UNIQUE INDEX `User_dni_key` ON `User`(`dni`);
```

No backfill statement — legacy rows keep `dni = NULL` and coexist under the unique index (safe on MySQL, per the finding above). No data migration, no `NOT NULL`.

## 2. DTO Changes

### `server/src/users/dto/create-user.dto.ts`

- **Add** `dni` as required:
  ```ts
  @IsString()
  @IsNotEmpty()
  dni: string;
  ```
- **Tighten** `nombre` and `apellido` from optional to required — replace each `@IsOptional() @IsString()` block with:
  ```ts
  @IsString()
  @IsNotEmpty()
  nombre: string;

  @IsString()
  @IsNotEmpty()
  apellido: string;
  ```
  (Drop the `?` optional markers; remove the now-unused `IsOptional` import only if no other field uses it — `activo` still uses it, so keep the import.)

### `server/src/users/dto/update-user.dto.ts`

- **Add** `dni` as required (DNI is editable and must stay non-empty on edit):
  ```ts
  @IsString()
  @IsNotEmpty()
  dni: string;
  ```
  Rationale: the proposal makes DNI editable and required in both flows. Making it required (not `@IsOptional`) in `UpdateUserDto` means a PATCH that omits `dni` is a 400 — consistent with the inline edit form which always submits `dni`. `username` remains absent (immutability unchanged).
- **Tighten** `nombre`/`apellido` the same way as create — from `@IsOptional() @IsString()` to `@IsString() @IsNotEmpty()`, required. `IsOptional` import stays (used by `rol`, `password`, `activo`).

> Note: `password` in `UpdateUserDto` stays `@IsOptional()` (empty = untouched) — unchanged.

## 3. Service Changes (`server/src/users/users.service.ts`)

### New constant + refined error discrimination

Add alongside `DUPLICATE_USERNAME_ERROR`:

```ts
const DUPLICATE_DNI_ERROR = 'El DNI ya está registrado.';
```

The existing `isUniqueConstraintError` collapses every `P2002` into "username" — with two unique columns that would mislabel a DNI collision. Replace/supplement it with a target-aware helper so the `P2002` backstop reports the right field:

```ts
function uniqueTargetIncludes(error: unknown, field: string): boolean {
  if (!(error instanceof Prisma.PrismaClientKnownRequestError) || error.code !== 'P2002') {
    return false;
  }
  const target = error.meta?.target;
  if (typeof target === 'string') return target.includes(field);
  if (Array.isArray(target)) return target.some((t) => String(t).includes(field));
  return false;
}
```

(`error.meta.target` on MySQL is the constraint/index name, e.g. `User_dni_key` / `User_username_key`, so `.includes('dni')` / `.includes('username')` discriminates reliably. Keep `isUniqueConstraintError` too if you prefer a generic fallback, but the two-field version is what the catch blocks use.)

### `create(dto, creadoPorId)` — add DNI pre-check + backstop

- After the existing `username` `findUnique` pre-check, add a symmetric DNI pre-check:
  ```ts
  const existingDni = await this.prisma.user.findUnique({ where: { dni: dto.dni } });
  if (existingDni) {
    throw new ConflictException(DUPLICATE_DNI_ERROR);
  }
  ```
- Add `dni: dto.dni` to the `data` object of `prisma.user.create`.
- In the `catch`, discriminate the `P2002` source:
  ```ts
  if (uniqueTargetIncludes(error, 'username')) {
    throw new ConflictException(DUPLICATE_USERNAME_ERROR);
  }
  if (uniqueTargetIncludes(error, 'dni')) {
    throw new ConflictException(DUPLICATE_DNI_ERROR);
  }
  throw error;
  ```

### `update(id, dto, callerId)` — add exclude-self DNI pre-check

This is the new consideration (`username` was never editable). After the existing `findUnique({ where: { id } })` existence check and the `activo` guards, before building `data`:

```ts
// findUnique can't express NOT — dni IS unique, but we must allow the row
// being edited to keep its own dni. findFirst with NOT: { id } does that.
const dniOwner = await this.prisma.user.findFirst({
  where: { dni: dto.dni, NOT: { id } },
});
if (dniOwner) {
  throw new ConflictException(DUPLICATE_DNI_ERROR);
}
```

Key point: **`findUnique` cannot take a `NOT` clause** (its `where` accepts only unique selectors), so the exclude-self pre-check MUST use `findFirst({ where: { dni, NOT: { id } } })`. This lets "edit a user without changing their DNI" pass (the only match would be the row itself, which `NOT: { id }` excludes), while still blocking a DNI that belongs to a *different* user.

- Add `dni: dto.dni` to the `data` object of `prisma.user.update`.
- Add the same `P2002` discrimination in a `try/catch` around `prisma.user.update` as the concurrency backstop (the pre-check is again non-atomic). Mirror the create catch: `uniqueTargetIncludes(error, 'dni')` → `DUPLICATE_DNI_ERROR`.

> `dto.dni` is required in `UpdateUserDto`, so it is always defined here. If a future change makes it optional again, guard both the pre-check and the `data` assignment with `if (dto.dni !== undefined)`.

## 4. New Frontend Page — `client/app/(dashboard)/usuarios/nuevo/page.tsx`

`'use client'` component. Auto-inherits `(dashboard)/layout.tsx` (Sidebar + Header) by route position — no layout code.

**State** (all `useState`):
- `form`: `{ username, dni, nombre, apellido, rol: UserRol, activo: boolean, password, confirmPassword }` — all strings except `rol`/`activo`; defaults `rol: 'empleado'`, `activo: true`, rest empty.
- `formError: string`, `submitting: boolean`.
- `router = useRouter()` from `next/navigation`.

**Fields** (reuse the EXACT Tailwind classes from `usuarios/page.tsx`'s inline form so the two look identical — do not reinvent):
- Wrapper card: `rounded-xl border border-stone-200 bg-white p-6 shadow-sm`.
- Form grid: `grid grid-cols-1 gap-4 sm:grid-cols-2`.
- Each field: `<div className="space-y-1">` with `<label className="text-sm font-medium text-stone-700">` + input.
- Input class (verbatim): `block w-full rounded-lg border border-stone-200 bg-stone-50 px-3 py-2 text-stone-900 focus:border-rose-400 focus:bg-white focus:outline-none focus:ring-2 focus:ring-rose-100`.
- Fields in order: **Usuario** (`username`, text, `required`, `autoComplete="off"`) — NOT disabled here (create allows setting it) · **DNI** (`dni`, text, `required`) · **Nombre** (`nombre`, text, `required`) · **Apellido** (`apellido`, text, `required`) · **Rol** (`<select>` over `USER_ROLES` with `ROLE_LABELS`, same select class + `onChange={(e) => updateField('rol', toUserRol(e.target.value))}`) · **Contraseña** (`password`, type `password`, `required`, `autoComplete="new-password"`) · **Repetir contraseña** (`confirmPassword`, type `password`, `required`, `autoComplete="new-password"`) · **Usuario activo** (checkbox, same `h-4 w-4 rounded border-stone-300 text-rose-500 focus:ring-rose-400` class + gap-2 row).
- Error banner (verbatim): `sm:col-span-2 flex items-center gap-2 rounded-lg border border-red-200 bg-red-50 px-4 py-3 text-sm text-red-600`.
- Actions row `flex gap-3 sm:col-span-2`: submit button (rose→red gradient, verbatim from list page, label `submitting ? 'Creando...' : 'Crear usuario'`) + **Cancelar** button (`<Link href="/usuarios">` styled with the outline button class `rounded-lg border border-stone-200 px-4 py-2.5 text-sm font-semibold text-stone-600 transition-all hover:bg-stone-50`).
- Header above card: reuse the `h1`/`p` block from the list page ("Nuevo usuario" / short subtitle).

**Client-side validation** (in `handleSubmit`, before the fetch — matches the "UX guard, not security" framing):
- Trim-check required fields: `username`, `dni`, `nombre`, `apellido`, `password`. If any empty → set `formError` (e.g. "Completá todos los campos obligatorios.") and return. (Native `required` covers the happy path; the JS check gives a consistent banner message and guards trimmed-whitespace.)
- Password match: `if (form.password !== form.confirmPassword)` → `setFormError('Las contraseñas no coinciden.')` and return.
- Optionally disable the submit button while passwords are non-empty and mismatched (mirrors proposal "submit enabled only when match").

**Submit**:
```ts
const payload: CreateUserPayload = {
  username: form.username,
  dni: form.dni,
  nombre: form.nombre,
  apellido: form.apellido,
  rol: form.rol,
  activo: form.activo,
  password: form.password,
};
await createUser(payload);      // confirmPassword is NEVER included
router.push('/usuarios');       // back to list on success
```
Wrap in `try/catch/finally` identical to the list page: `catch` sets `formError` from `err instanceof Error ? err.message : 'No se pudo conectar con el servidor.'`; `finally` clears `submitting`. Backend 400/409 (including `DUPLICATE_DNI_ERROR`) surface in the same red banner.

**confirm-password never leaves the browser** — it exists only in `form` state and the equality check; it is not part of `CreateUserPayload` and not sent.

## 5. Changes to `client/app/(dashboard)/usuarios/page.tsx`

- **Remove the create branch.** Delete `openCreateForm`; the inline form now only renders for edit. Simplify so `formOpen` is effectively "edit open": keep `editingUser`/`form`/`openEditForm`/`closeForm`, but the create path (`editingUser === null` submit branch, `EMPTY_FORM` for create, the `else` in `handleSubmit`) goes away. `handleSubmit` keeps only the `updateUser` branch.
- **Rewire "Nuevo usuario"** from a `<button onClick={openCreateForm}>` to a **`<Link href="/usuarios/nuevo">`** styled with the existing gradient button classes.
  - **Decision: `<Link>`, not `router.push`.** Rationale: this is pure navigation with no side effect to run first — `<Link>` is the semantically correct primitive, gives Next.js prefetching, works as a real anchor (middle-click/open-in-new-tab, keyboard, SSR-safe) and needs no `useRouter`/`onClick` handler. `router.push` is reserved for the create page's *post-submit* redirect, where navigation is a consequence of an async action. Keep the `!formOpen &&` guard so the link hides while the inline edit form is open (preserves current UX).
- **Add `dni` to the edit `FormState`** (`dni: string`) and to `EMPTY_FORM`/`openEditForm` pre-fill (`dni: user.dni ?? ''`). Add a **DNI input** to the inline edit form (same field markup/classes as the others, `required`), placed right after Usuario.
- **Add a DNI column to the table**: a `<th>` "DNI" header (same `<th>` class) after the Usuario column, and a matching `<td className="px-4 py-3 text-sm text-stone-600">{user.dni || '—'}</td>` cell in each row.
- The `updateUser` payload gains `dni: form.dni`.

## 6. Changes to `client/app/lib/users.ts`

- `UserListItem`: add `dni: string | null;` (list rows may include legacy rows with null DNI). Note `UserListItem extends UserData` and `UserData` has no `dni`, so `dni` is declared directly on `UserListItem`.
- `CreateUserPayload`: add `dni: string;` (required).
- `UpdateUserPayload`: add `dni: string;` (required — the inline edit form always sends it).

No change to the fetch functions themselves — they already serialize the whole payload.

> Server-side, add `dni: true` to the `USER_SELECT` whitelist in `users.service.ts` so the API actually returns `dni` in list/create/update responses (otherwise the new column would be selected out and never reach the client).

## 7. Alternatives Considered

- **Dedicated page vs. modal/inline for create** — the proposal already decided "dedicated page". Recorded rationale for a future reader: a dedicated route gets a shareable/bookmarkable URL, inherits the dashboard layout for free, and cleanly separates the now-heavier create form (DNI + confirm-password + required fields) from the list, without adding a modal dependency the project doesn't have (no headless-ui/radix in `package.json`). Not re-litigated here.
- **DB `@unique` vs. app-layer-only for `dni`** — covered above under the unique-constraint decision. The engine finding (MySQL allows multiple nulls) makes `@unique` the strictly-better choice; app-layer-only is documented as the rejected, engine-specific fallback.
- **`findFirst({ NOT })` vs. compare-then-skip in code** — could instead `findUnique({ where: { dni } })` and skip if `result.id === id`. Rejected: `findFirst({ where: { dni, NOT: { id } } })` expresses the intent in one query and can't be defeated by a forgotten `.id` comparison; it's also the idiomatic Prisma exclude-self pattern.
