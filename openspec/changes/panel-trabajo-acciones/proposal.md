# Proposal: Card-level actions (Iniciar / Editar / Desactivar) on `/ordenes-trabajo/panel`

## Intent
The Panel de Trabajo (`/ordenes-trabajo/panel`) answers "what is the state of the shop right now?" — a filter-reactive stats row, an estado-grouped Kanban board, and a per-mechanic workload section. But it is a **read-only lens**: a supervisor who spots a stuck or ready-to-start order on the board cannot act on it there. They must leave the panel, go to the list page (`/ordenes-trabajo`), find the same order again, and only then start it, edit it, or deactivate it. The panel makes the problem *visible* but forces a context switch to *do anything about it* — exactly the "act on it" workflow the sibling `panel-trabajo` change explicitly deferred (its Known Gaps: "A supervisor seeing a stuck order on the board still switches to the list/detail flow to change its estado; wiring board-side actions is a later change.").

This change is that deliberate follow-up. It adds three actions directly to each Kanban card, reusing the exact semantics already proven on the list page:

1. **Iniciar** — the same dual-behavior action as the list page's `IniciarTrabajoButton`: on a `pendiente` order it calls the atomic `iniciar` cascade then navigates to the work page; on an order already past `pendiente` (`en_proceso`, `terminado`) it is pure navigation with no API call (a second `iniciar` would 409). No confirmation dialog — byte-identical UX to today.
2. **Editar** — a plain `<Link href={`/ordenes-trabajo/editar/${id}`}>`, identical to the list page's Editar link.
3. **Desactivar** — the same `showConfirm(...)` → `updateOrdenTrabajo(id, { ...full payload, activo: false })` → `showSuccess`/`showError` flow as the list page's `AccionesMenu.handleToggleActivo`, resending the full order object with only `activo` flipped.

Success looks like: a supervisor working from the panel can start, edit, or deactivate an order in place, without ever leaving the board — and the board stays consistent by re-fetching the whole filtered panel after each action, exactly as the rest of the codebase does after any mutation.

## Scope

### In Scope
- **Interactive `KanbanCard` in `client/app/(dashboard)/ordenes-trabajo/panel/KanbanBoard.tsx`.** The currently pure-presentational card (`KanbanBoard.tsx:65-108`, zero event handlers) gains an actions cluster with the three actions above. All new UI, handlers, and the actions component are declared **inside the panel's own files** (`KanbanBoard.tsx` and/or a new sibling file under `panel/`), re-implementing a lighter equivalent of the list page's `AccionesMenu`/`IniciarTrabajoButton` rather than importing them — per the established D7 "re-declare per surface, don't extract a shared module" convention already applied twice on this page (see D3).
- **`onActionSuccess`-style callback wiring** from `panel/page.tsx` down through `<KanbanBoard>` → `KanbanColumn` → `KanbanCard`, bound to the existing `loadPanel()` re-fetch (`panel/page.tsx:98-122`). After any of the three actions succeeds, the whole filtered panel response is re-fetched — no optimistic/local-only bucket update (see D4). This mirrors the list page's `onIniciado={loadOrdenes}` / `onToggled={loadOrdenes}` convention.
- **Reuse of already-implemented client API functions** in `client/app/lib/ordenes-trabajo.ts`, unchanged: `iniciarOrdenTrabajo(id)` (`:138`), `updateOrdenTrabajo(id, data)` (`:123`), and the alert helpers `showConfirm`/`showSuccess`/`showError` from `client/app/lib/alerts.ts`. No new client API surface is introduced — the Kanban card already holds the full `OrdenTrabajoListItem` from the panel's fetched `data`, so the Desactivar payload is assembled from data already in hand.
- **Iniciar visibility rule:** the Iniciar action renders on cards in the `pendiente`, `en_proceso`, and `terminado` columns, and does **not** render on `cancelado` cards (see D1).
- **Spec reconciliation delta (mandatory, executed by `sdd-spec` — see the dedicated section below).** The sibling `panel-trabajo` change's ratified requirement "Board Is Read-Only With No New Drag-and-Drop Dependency" contains the phrase "or any other write interaction", which these three actions directly contradict. This change MUST narrow that requirement to drag-and-drop specifically and ADD a new requirement permitting the three named actions. This is an explicit, non-optional task for `sdd-spec`.

### Out of Scope / Non-Goals
- **No drag-and-drop.** `panel-trabajo`'s D2 had two distinct sub-decisions: (a) "no drag-to-change-estado / no DnD library" and (b) "no other write interaction". This change supersedes **only (b)**; (a) stays fully intact. No drag library is added, no drag-to-change-estado is introduced, cards still do not reorder on drag. See D2.
- **No change to the list page** (`client/app/(dashboard)/ordenes-trabajo/page.tsx`). Its `AccionesMenu`/`IniciarTrabajoButton` are read-only reference implementations to mirror; the file MUST NOT be modified by this change (D7 convention, applied by both sibling changes). No shared component is extracted.
- **No backend change of any kind.** `POST /ordenes-trabajo/:id/iniciar`, `PATCH /ordenes-trabajo/:id`, their DTOs, guards (`JwtAuthGuard` only), and the `iniciar` cascade/409-guard semantics are reused completely unmodified. This change is 100% frontend. See D5.
- **No new validation, permission, or guard changes.** Same access posture as the rest of the module — any authenticated user can act, deferred to the future Permisos feature.
- **No change to the other panel sections.** The stats/summary row, the filter bar, the "Carga por mecánico" section (`MecanicosWorkload`) from the sibling change, and the `GET /ordenes-trabajo/panel` + `GET /ordenes-trabajo/panel/mecanicos` endpoints are untouched. Only `KanbanCard` (and the callback threaded to it) changes.
- **No optimistic UI / local splice.** State reconciliation after an action is a whole-panel re-fetch only (D4).
- **No new Desactivar warning UX or vanish animation.** Beyond the existing confirm dialog and success toast, no extra visual warning or transition is added when a deactivated card disappears from the board (D6).
- **No minimal-PATCH convention.** Desactivar resends the full order object, matching the existing `handleToggleActivo` convention byte-for-byte; no new `{ activo: false }` partial-PATCH pattern is introduced (D8).

## Capabilities
### New Capabilities
- None (no new capability boundary; this extends the existing `ordenes-trabajo-panel` capability).

### Modified Capabilities
- `ordenes-trabajo-panel`: the Kanban board gains three per-card actions — **Iniciar** (dual-behavior, reusing the atomic iniciar cascade + work-page navigation), **Editar** (plain link to the edit route), and **Desactivar** (confirm → full-object `activo: false` PATCH → toast). The board transitions from read-only to interactive for these three named actions. Post-action state is reconciled by re-fetching the whole filtered panel via `loadPanel()`. The board's no-drag-and-drop posture is unchanged. The "Board Is Read-Only With No New Drag-and-Drop Dependency" requirement is narrowed to drag-and-drop specifically (see Spec Reconciliation Plan).

## Approach
**Frontend-only, re-implement per D7 (exploration's Approach 1).** The exploration weighed two paths: (1) re-declare a lighter actions cluster inside the panel's own files, or (2) extract `AccionesMenu`/`IniciarTrabajoButton` into a shared module imported by both the list page and the panel. Approach 1 is chosen decisively:

- **Rejected: extract a shared component (Approach 2).** It would require editing the untouched list page (`page.tsx`), directly contradicting every sibling change's explicit D7 decision, breaking a precedent set twice on this exact page, and enlarging the review surface. The single-source-of-truth benefit does not outweigh destabilizing the list page's proven UI for a change scoped to the panel.
- **Chosen rationale:** re-declaring a compact actions cluster scoped to `KanbanBoard.tsx`'s own files is fully consistent with the D7/§2.3 duplication convention (the panel already re-declares `ESTADO_LABELS`, `PRIORIDAD_*`, `formatFecha`, `mecanicoLabel` rather than importing them), carries zero risk to the untouched list page, can be tailored to the Kanban card's tighter real estate, and keeps `sdd-design` small. The one accepted cost is a third copy of "Editar link + conditional Iniciar + Desactivar-with-confirm" — mitigated by copying the `iniciar` dual-behavior logic carefully (see Risks).

**Iniciar — replicate the dual behavior exactly.** The card's Iniciar handler MUST reproduce `IniciarTrabajoButton`'s branch (`page.tsx:112-130`): if `orden.estado === 'pendiente'`, call `iniciarOrdenTrabajo(id)`, then on success run `onActionSuccess()` (re-fetch) and navigate to `/ordenes-trabajo/${id}/trabajo`; for any other rendered estado, skip the API call and navigate directly. The `pendiente`-only API call is what avoids the 409 the backend raises for an already-started order. No confirmation dialog — matching today's UX. Errors surface via `showError`, identical to the list page.

**Desactivar — reuse the full-object-resend flow.** The handler mirrors `handleToggleActivo` (`page.tsx:210-253`): `showConfirm({ title: 'Desactivar orden', ..., confirmButtonColor: '#e11d48' })`, and on confirm call `updateOrdenTrabajo(id, { fechaIngreso, kilometros, prioridad, motivoIngreso, estado, clienteId, vehiculoId, mecanicoId, tipoServicioIds, activo: false })` assembled from the card's `OrdenTrabajoListItem`, then `showSuccess`/`showError` and `onActionSuccess()`. Because the panel forces `activo: true` unconditionally, the re-fetch drops the just-deactivated card from the board entirely (D6).

**Editar — plain link.** A `<Link href={`/ordenes-trabajo/editar/${orden.id}`}>`, no confirm-before-navigate gate despite the panel's richer filter state (the exploration flagged and dismissed this; it is a plain, unremarkable link everywhere else in the app).

**UI layout — icon-triggered dropdown, re-implemented (see D9).** To fit the Kanban card's tighter real estate, the actions surface as a compact icon-triggered dropdown menu mirroring `AccionesMenu`'s pattern (re-implemented in the panel's files, not imported), rather than inline buttons crowding every card. The precise trigger affordance, menu positioning, and whether Iniciar sits inside the menu or as a standalone accent button are a design detail deferred to `sdd-design`; this proposal fixes the *dropdown-over-inline* direction and leaves the fine-grained layout open.

## Decisions (documented for later review/override)
- **D1 — Iniciar renders on `pendiente`, `en_proceso`, and `terminado` cards; never on `cancelado`.** *Rationale:* fixed product decision. This is a deliberate product call, **not** merely derived from the backend's 409 guard (which only blocks the API call for non-`pendiente`, not the navigation): a cancelled order has nothing to work on, so offering "Iniciar" (which for non-`pendiente` estados is pure navigation to the work page) there would be meaningless. On `en_proceso`/`terminado` cards Iniciar is pure navigation; on `pendiente` cards it runs the cascade first.
- **D2 — Supersedes only the "no other write interaction" half of `panel-trabajo`'s D2; the no-DnD half stays.** `panel-trabajo`'s D2 was a deliberate "not yet" for v1 that bundled two things: no drag-and-drop, and no write interactions at all. This change is the deliberate "now yes" for card-level actions, explicitly overriding the "no write interaction" framing — **not** an oversight or contradiction. The DnD-specific sub-decision (no drag library, no drag-to-change-estado) remains fully in force. *Rationale:* fixed product decision; the panel was always intended to grow an "act on it" workflow (the sibling change said so in its Known Gaps), and this is that follow-up.
- **D3 — Actions cluster re-implemented in the panel's own files, list page untouched.** *Rationale:* fixed decision continuing the D7 convention used by both sibling changes; re-declaration over shared-module extraction avoids editing the proven list-page UI and keeps the change's blast radius inside `panel/`.
- **D4 — Post-action refresh is a whole-panel re-fetch via `loadPanel()`, uniformly for all three actions.** No optimistic/local-only state update. *Rationale:* fixed product decision matching the codebase-wide "re-fetch the whole set after any mutation" convention (the list page never splices/optimistically updates); the stats row, board buckets, and cap signal all stay consistent because they come from one fresh response.
- **D5 — Zero backend surface; existing endpoints reused unmodified.** `iniciar` (`POST /ordenes-trabajo/:id/iniciar`) and `update` (`PATCH /ordenes-trabajo/:id`) plus their DTOs and `JwtAuthGuard` are already generic enough for this new caller. *Rationale:* the change is purely a new UI trigger wired to already-shipped API client functions; introducing backend changes would expand scope and risk for no benefit.
- **D6 — Deactivated card disappears from the board on re-fetch; no extra warning or animation.** The panel's `buildPanelOrdenTrabajoWhere` forces `activo: true` with no override, so after the confirm dialog and re-fetch the card simply vanishes — accepted as correct, not a bug, and needing no special transition. *Rationale:* fixed product decision; the existing confirm dialog is a sufficient guard, and the panel (unlike the list page's in-place "Inactiva" pill) has no way to show deactivated orders, so vanishing is the only coherent outcome.
- **D7 — List page (`ordenes-trabajo/page.tsx`) MUST NOT be modified.** *Rationale:* fixed decision; it is the read-only reference for the exact behavior to mirror, per every sibling change's D7 "list page untouched" convention.
- **D8 — Desactivar resends the full order object (`{ ...allFields, activo: false }`), byte-identical to `handleToggleActivo`.** No minimal `{ activo: false }` PATCH is introduced even though the DTO's `@IsOptional()` fields would technically accept one. *Rationale:* fixed product decision; the card already has the full `OrdenTrabajoListItem` in hand, and matching the existing full-resend convention keeps one convention across surfaces rather than introducing a second partial-PATCH pattern to reason about.
- **D9 — Icon-triggered dropdown over inline buttons (fine layout deferred to `sdd-design`).** *Rationale:* Kanban cards have tighter real estate than a table row or list-page tarjeta; a dropdown keeps the card compact and mirrors the list page's `AccionesMenu` mental model. The exact trigger/positioning/standalone-Iniciar details are a `sdd-design` decision — flagged, not silently left open.

## Spec Reconciliation Plan (explicit task for `sdd-spec`)
The requirement **"Board Is Read-Only With No New Drag-and-Drop Dependency"** currently lives in the sibling `panel-trabajo` change's delta at `openspec/changes/panel-trabajo/specs/ordenes-trabajo-panel/spec.md:206-220` (it merges to `openspec/specs/ordenes-trabajo-panel/spec.md` once `panel-trabajo` is archived — that merged path does not exist on disk yet because the sibling changes are unarchived). Its text says: *"Kanban cards MUST NOT support drag-to-change-estado **or any other write interaction**..."* The phrase "or any other write interaction" directly contradicts this change's three actions.

`sdd-spec` MUST, in this change's delta:
1. **MODIFY** that requirement to narrow its scope to drag-and-drop specifically — e.g. retitle to something like "Board Has No Drag-and-Drop; Explicit Action Controls Are Permitted" and drop/replace the "or any other write interaction" clause. Its two existing scenarios ("Dragging a card does not change its estado" and "No drag-and-drop dependency is introduced") are already DnD-only and remain true verbatim — they MUST be preserved.
2. **ADD** a new requirement explicitly permitting the three named card actions (Iniciar with its dual behavior + `cancelado` exclusion per D1, Editar as navigation, Desactivar with confirm + full-object `activo: false` PATCH), with scenarios covering: Iniciar on a `pendiente` card runs the cascade then navigates; Iniciar on a non-`pendiente` non-`cancelado` card is pure navigation with no API call; Iniciar is absent on `cancelado` cards; a successful action triggers a whole-panel re-fetch; and a deactivated card disappears from the board after re-fetch.

This is a MODIFIED + ADDED delta pair, not a silent contradiction. Because `panel-trabajo` is not yet archived, `sdd-spec` should author this change's delta against the requirement as it stands in the sibling change's delta file, and the archive tooling will reconcile the merge order.

## Rollback Plan
This change modifies the frontend only — no schema, migration, backend, or data change — so rollback carries no data-loss risk. Unlike the two prior purely-additive panel changes, this one **modifies existing behavior**: it makes the read-only board interactive. The rollback path is therefore "revert to the read-only `KanbanCard`", not just "delete new files". Rollback is mechanical:
1. Revert the PR, or piecemeal: restore `KanbanCard` (and `KanbanColumn`/`KanbanBoard`) to its pure-presentational form (`KanbanBoard.tsx:65-155` as it stands today — no event handlers, props `{ orden }` only), and remove any new actions component/file added under `panel/`.
2. Remove the `onActionSuccess` callback threading from `panel/page.tsx` and the `<KanbanBoard>`/`KanbanColumn`/`KanbanCard` prop chain, leaving `loadPanel()` and every other section untouched.
3. Revert this change's spec delta (the MODIFIED narrowing + the ADDED actions requirement), restoring the "Board Is Read-Only With No New Drag-and-Drop Dependency" requirement to its original wording.

No client API function, backend endpoint, DTO, or the list page is modified, so reverting cannot affect them. `iniciarOrdenTrabajo`/`updateOrdenTrabajo`/`getOrdenTrabajo` are pre-existing and shared with the list page — they are only *called* by the new handlers, never changed, so they stay in place.

## Known Gaps / Accepted Tradeoffs
- **A third copy of the actions logic (D3).** The Editar link + conditional Iniciar + Desactivar-with-confirm logic now exists in three places (table `AccionesMenu`, card-view button, and the new Kanban cluster). Accepted per the D7 convention; the drift risk is concentrated in the `iniciar` dual-behavior branch (see Risks) and mitigated by careful copying. Promotion to a shared `ordenesActions` module remains a clean future refactor to do when all surfaces can be touched together.
- **Whole-panel re-fetch after each action (D4).** Every successful action re-issues `GET /ordenes-trabajo/panel`. This is heavier than a local splice but is the codebase convention and guarantees the stats row, board buckets, and cap signal stay consistent. Acceptable given the bounded (capped) board size.
- **Deactivated cards vanish (D6).** More visually disruptive than the list page's in-place "Inactiva" pill, but the only coherent outcome given the panel's forced `activo: true` filter; accepted with no special UX.
- **No confirm-before-navigate on Editar despite richer panel filter state.** Navigating to the edit route loses the panel's five-control filter state on return; accepted as a plain link, consistent with every other edit link in the app.
